# ADR-0004: Use Alembic for Migrations

## Status

Accepted

## Date

2026-07-19

## Context

Production schema changes need:

- Version history
- Upgrade paths
- Downgrade support
- Reviewable migration files
- Deployment-safe execution

## Decision

Use Alembic as the only supported database schema migration tool.

- SQLAlchemy metadata describes the intended schema.
- Alembic migrations apply schema changes.
- `Base.metadata.create_all()` must not replace migrations.
- Autogenerated migrations must be reviewed manually.

## Consequences

### Positive

- Reviewable schema history
- Repeatable deployments
- Upgrade and downgrade workflows
- Native SQLAlchemy integration
- Better control over destructive changes

### Negative

- Migrations require manual review
- Downgrades are not always safe
- Data migrations may need special handling
- Developers must keep models and migrations aligned

## Alternatives Considered

### `Base.metadata.create_all()`

It can create missing tables but does not provide a migration history or safe schema evolution.

### Manual SQL files only

Manual SQL files provide control but lose Alembic autogeneration and metadata integration.

### Application startup migrations

Running migrations from every application process creates deployment and concurrency risks.
