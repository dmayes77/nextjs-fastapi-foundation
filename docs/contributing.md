Contributing

Purpose

This guide explains how to make changes to the repository safely and consistently.

The project is designed to become a reusable GitHub template, so contributions must preserve clarity, simplicity, and predictable setup.

Before Starting

Before changing code:

1. Read README.md.
2. Read the relevant document in docs/.
3. Confirm the working tree is clean.
4. Pull the latest main branch.
5. Create a focused feature branch.
6. Understand the checkpoint and acceptance criteria.

Do not begin unrelated work in the same branch.

Branch Naming

Use short, descriptive branch names.

Recommended patterns:

feature/frontend-health-page
feature/fastapi-database-foundation
fix/openapi-client-generation
docs/update-local-setup
test/add-todo-route-coverage
chore/update-tooling

Avoid vague names such as:

changes
updates
work
test-branch

Commit Messages

Use this format:

type(scope): description

Common types:

- feat
- fix
- test
- docs
- chore
- refactor
- ci

Examples:

feat(frontend): add health status page
feat(api): add readiness endpoint
feat(database): configure async SQLAlchemy
fix(api-client): preserve camelCase response fields
test(api): cover missing Todo responses
docs: explain database setup
ci: add PostgreSQL integration job

Keep commit messages:

- Short
- Specific
- Written in the present tense
- Focused on one logical change

Change Scope

A contribution should solve one clear problem.

Good examples:

- Add the Next.js scaffold
- Add the FastAPI scaffold
- Configure SQLAlchemy
- Add Alembic
- Add the Todo feature
- Add frontend tests

Avoid combining unrelated work such as:

- Database setup
- UI redesign
- Authentication
- Deployment configuration

inside one pull request.

Development Workflow

The standard workflow is:

1. Create a branch.
2. Make the smallest complete change.
3. Run the relevant checks.
4. Review the diff.
5. Commit the change.
6. Push the branch.
7. Open a pull request.
8. Wait for required checks.
9. Address review feedback.
10. Merge only when the branch is ready.

Required Validation

Before committing, run the smallest relevant validation.

Before opening a pull request, run the full safe validation command once it exists:

pnpm check

A change is not complete when:

- Tests are failing
- Lint is failing
- Generated API files are stale
- Migrations are missing
- Documentation is inaccurate
- Environment examples are incomplete

Frontend Contributions

Frontend changes should:

- Use Server Components by default
- Keep Client Components focused
- Use the approved API client
- Avoid direct PostgreSQL access
- Use the @/ import alias
- Include loading, empty, and error states when appropriate
- Add tests for meaningful interactive behavior

Do not hardcode backend URLs inside features.

Backend Contributions

Backend changes should:

- Use versioned API routes
- Keep routes small
- Put business logic in services
- Use async SQLAlchemy
- Use Pydantic request and response schemas
- Return standard errors
- Include request IDs
- Add tests
- Add migrations for schema changes

Do not use Base.metadata.create_all().

Database Contributions

When changing the schema:

1. Update SQLAlchemy metadata.
2. Generate an Alembic migration.
3. Read the migration.
4. Confirm constraint names.
5. Check upgrade behavior.
6. Check downgrade behavior when practical.
7. Run relevant tests.
8. Commit the model and migration together.

Never assume an autogenerated migration is correct without reviewing it.

API Contract Changes

When changing:

- Request fields
- Response fields
- Endpoint paths
- Query parameters
- Status codes

you must:

1. Update FastAPI schemas and routes.
2. Export OpenAPI.
3. Regenerate the frontend client.
4. Update frontend usage.
5. Run the client freshness check.
6. Update documentation when behavior changed.

Generated client files must not be edited manually.

Documentation

Update documentation when a contribution changes:

- Installation
- Commands
- Environment variables
- API behavior
- Database setup
- Migration workflow
- Testing
- Deployment
- Architecture
- Project structure

Every documented command must have been tested successfully.

Pull Request Description

Each pull request should explain:

Summary

What changed?

Reason

Why was the change needed?

Testing

Which commands or manual checks were completed?

Risks

What could break or require extra attention?

Screenshots

Include screenshots for visible UI changes when useful.

Review Checklist

Before requesting review, confirm:

- The branch contains only intended changes.
- The working tree is clean.
- Code follows repository standards.
- Tests pass.
- Lint passes.
- Formatting checks pass.
- Migrations are included when needed.
- Generated API output is current.
- Documentation is accurate.
- No secrets are committed.
- No personal machine paths are committed.

Secrets

Never commit:

- .env
- Database passwords
- Access tokens
- API keys
- Private certificates
- Production connection strings
- Personal credentials

If a secret is committed accidentally:

1. Stop.
2. Rotate the secret.
3. Remove it from the repository.
4. Review Git history.
5. Document the incident if necessary.

Deleting the visible line alone may not remove the secret from Git history.

Generated Files

Do not manually edit generated API client files.

Generated files must be reproducible from checked-in source files and documented commands.

When generated files are committed, CI should verify that they are current.

Dependency Changes

When adding a dependency:

1. Confirm an existing dependency does not already solve the requirement.
2. Prefer actively maintained packages.
3. Review license compatibility.
4. Avoid large packages for small utilities.
5. Update the appropriate lockfile.
6. Document setup when required.

Do not add dependencies only because they may be useful later.

Backward Compatibility

Changes to the reusable template should avoid unnecessary breaking changes.

Breaking changes must be:

- Intentional
- Documented
- Tested
- Included in release notes
- Reflected in the version number

Template Cleanliness

Because this repository becomes a GitHub template, contributions must avoid:

- Personal names in application code
- Personal email addresses
- Local absolute paths
- Private project identifiers
- Company-specific branding
- Real credentials
- Production data

Examples should remain generic and reusable.

Issues

A useful issue should include:

- Clear title
- Current behavior
- Expected behavior
- Steps to reproduce
- Relevant logs
- Environment details
- Screenshots when useful

Do not include secrets in issue reports.

Final Rule

Every contribution should leave the repository easier to understand, safer to reuse, and no more complicated than the requirement demands.
